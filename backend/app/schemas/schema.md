# Schema

## User

**Purpose:**
A dashboard user who owns projects.

**Columns:**

- id (UUID, PK, default gen_random_uuid())
- email (TEXT, UNIQUE, NOT NULL)
- password_hash (TEXT, NOT NULL)
- created_at (TIMESTAMPTZ, NOT NULL, default now())
- updated_at (TIMESTAMPTZ, NOT NULL, default now())

**password_hash:**
Kept for now. Only remove this if we make an explicit, separate decision to adopt Neon Auth (or another identity provider) as our auth system — that's a real architectural commitment, not implied by using Neon for Postgres hosting.

## Project

**Purpose:**
Represents one AI application.

**Columns:**

- id (UUID, PK, default gen_random_uuid())
- owner_id (UUID, FK → User.id, NOT NULL, ON DELETE CASCADE)
- name (TEXT, NOT NULL)
- description (TEXT, NULLABLE)
- api_key_hash (TEXT, UNIQUE, NOT NULL)
- created_at (TIMESTAMPTZ, NOT NULL, default now())
- updated_at (TIMESTAMPTZ, NOT NULL, default now())

## Memory

**Purpose:**
A piece of knowledge stored for a project.

**Columns:**

- id (UUID, PK, default gen_random_uuid())
- project_id (UUID, FK → Project.id, NOT NULL, ON DELETE CASCADE)
- external_id (TEXT, NULLABLE)
- content (TEXT, NOT NULL)
- metadata (JSONB, NULLABLE, default '{}')
- created_at (TIMESTAMPTZ, NOT NULL, default now())
- updated_at (TIMESTAMPTZ, NOT NULL, default now())

**Constraints:**

- UNIQUE (project_id, external_id) — external_id only needs to be unique within a project, not globally. Two different projects can each have their own `"user_123"`.

**external_id:**
Developer-defined identifier, set by the caller, not generated by us.

Example: `"user_123"`, `"conversation_456"`

Used so developers can map memories back to objects in their own system without exposing our internal UUIDs.

**metadata:**
Arbitrary developer-defined metadata. Not queried or validated by us in v1 — just stored and returned.

Example:

```json
{
  "source": "support-chat",
  "user_id": "123",
  "conversation": "conv_89"
}
```

## Embedding

**Purpose:**
Vector representation of a Memory. Kept separate from Memory so we can change embedding models or providers without touching the Memory table, and so a Memory can hold more than one embedding at once (e.g. during a model migration, or if comparing providers).

**Columns:**

- id (UUID, PK, default gen_random_uuid())
- memory_id (UUID, FK → Memory.id, NOT NULL, ON DELETE CASCADE)
- model (TEXT, NOT NULL)
- dimensions (INTEGER, NOT NULL)
- vector (VECTOR, NOT NULL)

**Constraints:**

- UNIQUE (memory_id, model) — one embedding per model per memory. Prevents duplicate rows from retries or bugs.

---

## Relationships

```
User (1) -----> (*) Project
Project (1) ---> (*) Memory
Memory (1) ----> (*) Embedding
```

A memory can have more than one embedding — one per model. In the MVP, in practice, there will only ever be one row per memory, but the relationship isn't constrained to 1:1 so we're not blocked when we add a second embedding model later.

**On delete:** all foreign keys cascade. Deleting a User deletes their Projects; deleting a Project deletes its Memories; deleting a Memory deletes its Embeddings. This is irreversible — no soft-delete in v1. Acceptable for now; revisit if accidental deletion becomes a real risk once there are paying customers.

**UUID generation:** database-side, via Postgres's built-in `gen_random_uuid()`. No UUIDs are generated in application code. IDs aren't known until after an insert returns.

## Deliberately NOT in MVP

- Organizations
- Teams
- RBAC
- Multiple API keys per project
- Billing
- Agent table
- Memory versioning
- Knowledge graph
- Cross-agent synchronization

Check this list before adding anything. If it's here, it doesn't get built until a real user asks for it.

---

## Design principles

- Keep the schema minimal.
- Every column must have a purpose today, not a hypothetical one.
- Every table must represent a real-world entity.
- Prefer JSONB over premature normalization.
- Add abstractions only when there is a concrete need.

---

Locked until the MVP is built. Do not add, remove, or rename columns before then.
